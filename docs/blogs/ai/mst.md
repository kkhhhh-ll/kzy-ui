## 1、什么是LLM？有什么特点？
```
LLM，大语言模型，通过海量文本数据训练的语言模型。
特点：
（1）无状态，导致对话状态的遗忘
（2）只能生成文本，无法操纵外部世界
（3）上下文窗口限制
（4）答案概率性生成
类似于人的“大脑”，会想，会说，但不会做。
```
## 2、什么是token？怎么生成？
```
token，词元，LLM处理文本的最小单元。
tokenization：BPE
```

## 3、什么是Agent？与LLM有什么关系？怎么去理解Agent Loop?
```
人工智能领域，agent智能体是指任何能通过传感器感知外部环境，并自主地通过执行器去采取行动的实体。
如果说LLM对应于人的“大脑”，只会想，只会说，那Agent就是承载LLM的人类，他能够将LLM的想法付诸于行动。
而Agent的运行机制，称之为Agent Loop，即通过一个“感知-思考-行动”的循环不断地处理问题。
通常需要实现：提示工程（语言）、MCP（Tool，USB）、Memory、Context、Skills（操作手册）、RAG
```

## 4、什么是提示工程？
```
提示工程就是agent与LLM沟通的语言。
目标：写出精确的提示词，提高LLM的运行效率与准确性。
技巧：
（1）赋予LLM角色，可以具体一点儿，比如说，你现在就是陶喆，请用陶喆的讲话方式跟我对话。
（2）少样本提示
（3）背景
（4）会话限制
（5）输出要求
（6）CoT，思维链，引导模型“一步一步”地思考
```
## 5、什么是记忆系统？为什么需要记忆系统？
```
人类的记忆是一个多层级的认知系统，不仅能存储信息，还可以通过时间、重要性和上下文对信息进行分类和整理。
从认知心理学可分为：感觉记忆，工作记忆，长期记忆。
记忆系统是人类智能的一个重要能力，人类可以通过在以往的经历中获取经验和教训，并将其应用于任何场景。
因此，可以得知Agent的记忆系统 是让Agent能够存储、保留和回忆信息的机制
为什么需要记忆系统？
LLM本身的局限性，LLM本身是无状态的，会导致对话状态的遗忘，引入记忆系统就是解决LLm本身的局限性。
```
## 6、什么是RAG系统？为什么需要RAG？
```
RAG是一种结合了信息检索和文本生成的技术。核心思想是，生成回答前，先从外部知识库中检索相关信息，并将检索到的信息作为上下文提供给大模型，让模型生成更准确、更具时效性的回答。
解决LLM知识局限性的问题？比如，LLM训练结束于一个月前，这一个月内发生的事情，模型无从得知。特定领域知识缺失
```

## 7、什么是MCP？什么是Skills？二者有什么区别？
```
MCP主要解决“连接”问题，是智能体与外部世界交互的标准化接口。
Skills主要解决“能力”问题，领域知识和工作流的封装。
区别：
用途：MCP类似于“USB”，它可以使agent获取大量的工具，但是并不知道如何使用。SKills类似于“操作手册”，可以教导agent完成相应的功能。
加载方式：智能体通过FastMCP连接外部库，获取的工具一时间被加载到智能体的上下文信息中，造成token的浪费或导致上下文窗口爆炸。Skills采用渐进式披露的机制加载资源，Skills一般分为元数据+body。agent开始运行时，通过skillsLoader获取所有skills的元数据并将其加载到上下文窗口中，但是skills的元数据要比一个工具小的多的多。

```

## 8、什么是上下文工程？为什么需要上下文工程？
```
上下文工程是在上下文窗口中为下一步填充恰到好处信息的精妙艺术与科学。
上下文腐蚀问题：随着上下文窗口中的tokens增加，模型从上下文中准确回忆信息的能力降低。
```
## 9、什么是harness engineer？为什么需要harness engineer？需要实现哪些能力？
```
LLM像是一匹很有能力的马，但是需要一套“马具”来稳定，可靠的驾驭其能力，使其从一个“引擎”变成一个能上路的车。
LLM自身的局限性
记忆层：解决模型无状态的问题，文件系统
执行层：解决模型操作外部世界的能力
反馈层：通过自动化测试等手段解决输出不可靠的问题 linter,playwright、ci/cd（输出不确定，但是验证是确定的）
编排层：解决复杂任务的拆解与协作问题
```

## 10、multi-agent?有什么区别？
```
Sub-Agents（外包模式）:解决信息太多的问题
1、各干各的，不沟通
2、每个子agent相互独立，都有自己的上下文
3、只返回结果，不反悔过程

Agent-terms（团队）：解决协作太复杂的问题
agent可以互相沟通，边干边沟通
```

## 11、乐观锁与悲观锁是什么？区别是什么？
```
乐观锁是假设数据不会冲突，更新时再检查版本号。不加锁，更新失败就重试。读多写少
悲观锁是假设数据一定会冲突，加锁，阻塞其他操作，直到事务结束。读少写多

```
## 12、Agent工作流程的编排
```
一、初始化阶段（构建能力基础）
准备好 Agent 运行所需的一切"装备"：
LLM 配置：选择模型、设置温度/最大 token 等参数
工具注册：加载 Agent 可调用的工具/函数
记忆加载：恢复历史会话、用户偏好、长期记忆
系统提示词：设置 Agent 的角色、行为规范、输出格式
会话上下文：初始化会话 ID、元数据等
二、运行阶段（循环执行直到任务完成）
这是一个感知 → 决策 → 执行 → 观察 → 反思的循环：
构建上下文
组装 system、user 消息、工具定义、历史记忆、检索到的知识
调用 LLM 决策
LLM 分析当前状态，决定下一步：直接回答 / 调用工具 / 继续思考
执行动作
若调用工具 → 执行工具，获取结果
若直接回答 → 进入结束流程
观察与整合
将工具执行结果作为新的上下文，追加到对话中
评估与反思
检查结果是否满足目标？是否需要继续？信息是否充足？
循环判断
满意 → 结束，输出最终答案
不满意 → 返回步骤 1，继续循环
超过最大步数 → 强制结束并提示

三、结束阶段（收尾工作）
更新记忆系统（存储本次对话的关键信息）
记录 token 使用量
持久化会话状态
```

## 13、怎么判断LLM可以结束本次回答了？

核心原则：让 LLM 自己判断是否完成最可靠，但必须有硬性兜底防止无限循环。

1. 达到最大步数 → 强制结束
   ↓
2. 检测到死循环 → 强制结束
   ↓
3. LLM 调用了工具 → 不能结束，继续
   ↓
4. LLM 没调用工具 → 检查回答质量
   ├─ 有意义 → 可以结束
   └─ 无意义 → 可能要继续或报错

## 14、如果Agent在某一部执行出现差错了，需要怎么办？

```
核心思想：先重试（临时错误）→ 再降级（工具失败）→ 最后终止（致命错误），永远给用户一个有意义的反馈。
步骤执行出错
    │
    ├─ 是临时性错误（网络超时、限流）？
    │   └─ 是 → 重试（指数退避）→ 成功则继续，失败则降级
    │
    ├─ 是工具执行错误？
    │   ├─ 重试3次 → 仍失败
    │   └─ 使用降级结果/跳过该工具 → 继续执行
    │
    ├─ 是LLM响应异常？
    │   ├─ 格式错误 → 重新生成（修正提示词）
    │   ├─ 内容为空 → 重试
    │   └─ 拒绝回答 → 返回友好提示
    │
    └─ 是致命错误（状态损坏、资源不足）？
        └─ 终止执行，返回错误提示

二、错误分类与处理策略
工具调用失败，比如API超时、网络错误，可以重试2-3次 → 降级 → 跳过。
LLM响应异常，比如格式错误、内容为空，可以重新请求 → 修正提示词
资源不足，比如，Token超限，可以压缩上下文 → 降级处理
```

## 15、上下文窗口爆炸了怎么办？
```
上下文工程为上下文窗口填充信息时，需要经过Gather(收集)->Select(选择)->Structure(结构化)->Compress(压缩)四步处理信息。
上下文工程中配置了窗口最大的token数量，压缩阈值，最小可接收系数
1、收集（系统、对话历史、用户查询、额外包）
2、选择（系统+用户查询保留。对对话历史和额外包里的信息计算相关性和时效性计算，计算最后的分数，按分数从高到低排序，按token余量去选择）
3、结构化：系统指令，当前任务，任务状态（完成，未完成），参考信息，对话历史，期望输出
4、压缩：
智能截断（必须保留的要留下来，其他的进行总结）->直接截断（保留后几轮对话）
```
